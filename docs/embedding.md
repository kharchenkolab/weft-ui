# Embedding & programmatic use

weft-ui is a local companion server (it runs with the user's SSH keys —
one process per workspace, bound to `127.0.0.1`). Beyond the standalone
app, it offers three integration surfaces for other applications:

1. **Deep links** — every screen has a URL; open one in a window or tab.
2. **Embedded panels** — iframe any screen chrome-less into your own UI.
3. **HTTP API** — the same tool facade and event stream the UI itself uses.

## 1. Deep links

The hash route is the source of truth for page, tab, and selection:

| Route | Shows |
|---|---|
| `#/jobs` | jobs table |
| `#/jobs/jb_<id>` | jobs table with a job selected |
| `#/jobs/kernels` · `#/jobs/kernels/krn_<id>` | kernels subtab (optionally one selected) |
| `#/jobs/services/svc_<id>` | services subtab |
| `#/jobs/envs/env:v1:<hash>` | environments subtab |
| `#/provenance/<job-id-or-ref>` | full provenance view for a target |
| `#/compute` · `#/compute/<site>` | site cards (optionally one open) |
| `#/chat` · `#/chat/c_<id>` | chat (optionally one conversation) |
| `#/activity` | audit trail |
| `#/wizard` | add-compute wizard |

Selection changes replace the history entry (arrow-key browsing doesn't
pollute history); page and tab changes push, so back/forward work.

A full URL needs the token (see [Auth](#auth)):

```
http://127.0.0.1:8999/?token=<token>#/jobs/kernels
```

## 2. Embedded panels

Add `embed=1` and the app renders the routed content without its nav
rail and topbar — the host supplies the chrome:

```html
<iframe src="http://127.0.0.1:8999/?token=<token>&embed=1#/jobs/kernels"
        style="width:100%;height:640px;border:0"></iframe>
```

Framing is opt-in. The server sends `Content-Security-Policy:
frame-ancestors 'self'` by default, so no other page can embed it. Allow
your host app's origin in the workspace config
(`<workspace>/.weft-ui.json`) and restart the server:

```json
{ "embed_origins": ["http://127.0.0.1:7777"] }
```

This does two things: the origin may frame the UI (CSP), and `/api`
requests carrying that `Origin` header pass the origin check.

Panels are live: they consume the same SSE event spine as the full app,
and multiple panels of the same workspace in one browser share a single
event connection (Web-Locks leader + BroadcastChannel).

Layout notes: panels are responsive — below ~900 px the side columns
slim down, below ~640 px detail panes stack. Any iframe size from a
sidebar sliver to a full window renders sensibly.

## 3. HTTP API

Everything the UI does goes through HTTP endpoints you can call
directly. Errors are payloads inside a 200 (weft's returns-never-raises
contract crosses HTTP intact); HTTP status codes are transport only.

**Tool facade** — every tool in weft's `PUBLIC_TOOLS`, generated by
introspection:

```sh
curl -X POST http://127.0.0.1:8999/api/w/jobs_where \
  -H "Authorization: Bearer <token>" -H "Content-Type: application/json" \
  -d '{"limit": 50}'
```

Destructive/account-level tools (`register_site`, `site_teardown`,
`gc_sweep`, `gc_packages`) return `409 {"consent_required": …}` until
repeated with `"_confirm": true`.

**Event stream** — one SSE stream with cursor replay:

```
GET /api/events?cursor=<last-seen-seq>   (Authorization: Bearer <token>)
```

Each event carries a monotonic `seq`; persist the last applied value and
reconnect from it. A `_resync` control event means your cursor is
unusable (wiped or ahead) — refetch lists and adopt the new seq.

**Chat** — conversations with the workspace agent:

```
POST /api/chat/conversations                     → {id, …}
POST /api/chat/conversations/{id}/message        {"text": …}
GET  /api/chat/conversations/{id}/stream         (SSE: replay then live)
POST /api/chat/conversations/{id}/approval       {"request_id", "decision"}
GET  /api/chat/setup                             skills / MCP servers / consent
```

A turn that hits the consent gate emits `approval_request` on the
conversation stream and pauses until the approval endpoint answers —
the same defer-and-resume the UI's card uses.

## Auth

Each server process mints a bearer token at startup (printed with the
URL), or accepts a fixed one via `--token`. Every `/api` request needs
it (`Authorization: Bearer …` or `?token=…`); the served page embeds it.
Cross-origin browser requests are additionally filtered by the `Origin`
allowlist (`embed_origins` + the server's own origins).

For a host app: run the server with a token you choose, or parse the
startup line. Treat the token like a session secret — anyone holding it
drives the workspace with the user's identities.

## Multiple workspaces

One server serves one workspace (a deliberate single-writer lock on
`.weft/ui.lock`). To embed panels for several workspaces, run one server
per workspace on distinct ports and point each panel at its server:

```sh
weft-ui serve --workspace ~/proj-a --token tokA --port 8901
weft-ui serve --workspace ~/proj-b --token tokB --port 8902
```

## Security posture (unchanged by embedding)

- Binds `127.0.0.1` only — panels work for host apps on the same
  machine (or via your own reverse tunnel, at your own risk).
- `frame-ancestors` and the Origin allowlist stay closed by default;
  `embed_origins` is the single opt-in.
- The process holds the user's SSH agent and workspace files. Do not
  expose it beyond the machine boundary.
