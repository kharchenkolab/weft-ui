Design notes

  1. Weft-blind panel + skill. The panel's contract is a stream of typed turn events (text_delta, tool_call, tool_result, approval_request, usage). Everything weft-specific enters through a renderer registry keyed by tool name — swap the skill and tools, the panel is unchanged. The mounted skills/weft/ is the agent's doctrine; the cards make it visible.
  2. Per-session model + budget. Model is a per-session setting; the meter is a live usage bar with a hard-stop cap. Respect-the-quota, surfaced not buried.
  3. Plan card = task_submit(dry_run=True). weft returns the plan before any effect; the renderer just lays it out. Nothing here is invented UI — these are the consent moments weft already designed.
  4. Placement with reasons. Ranked sites carry their why, rejected sites carry their cause (colored by error class). Policy notes are surfaced verbatim and shown entering the decision — here "prefer nights for >1 h jobs" is read against ~18-min elements and visibly set aside, not silently ignored.
  5. Consent tier from the plan. The gate reads the plan (staging bytes, env build, walltime), not the phrasing — free / costly / spending. This submission trips costly on staging size.
  6. One gate, two faces. Approve resumes the same agent turn (defer + resume). "Always allow under X" writes the workspace threshold that also governs the equivalent jobs-panel button — human and agent hit identical enforcement.
  7. Digest, not firehose. A 2,000-element array renders from coalesced array.progress counts + failure previews — a handful of lines. The panel never ingests per-element events.
  8. Shared error card. Byte-for-byte the same component as the jobs panel FAILED pane — the reference UI teaches by reuse. The remediation is the exact array_retry call, honoring never-size-down.
  9. Grades are information. The manifest chip expands to the per-component breakdown and notes/step_notes — the "why this rung" story, in the skill's tone (escape hatches are normal, supported moves).
  10. Cross-links both ways. Job/env ids and drefs in agent output linkify into the panels; the panels' "ask the agent" seeds a chat with the object. Seconds to decision, in either direction.
